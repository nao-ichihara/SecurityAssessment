# フレームワーク追加用プロンプト

---

## 使い方

このプロンプトをコピーし、`【】` で囲まれた箇所を追加したいフレームワークの情報に置き換えてAIに送信してください。  
2つのファイルを別々に依頼するか、続けて依頼するかはお好みで構いません。

---

## ファイル①：質問票 JSON

```
あなたはサイバーセキュリティの専門家です。
以下の仕様に従い、企業向けセキュリティリスクアセスメントシステム用の質問票 JSON ファイルを作成してください。

---

## 対象フレームワーク

- 名称：【例：PCI DSS v4.0】
- 対象：【例：クレジットカード業界のデータセキュリティ標準】
- セクション構成：【例：12の要件／6つのカテゴリ など】
- 想定質問数：【例：全体で 80〜100 問程度】

---

## 出力仕様

以下の JSON スキーマに厳密に従ってください。

```json
{
  "id": "【スネークケースの一意ID。例：pci_dss】",
  "label": "【表示名。例：PCI DSS v4.0】",
  "sublabel": "【副題。例：全12要件】",
  "frameworks": ["【フレームワーク名。例：PCI DSS v4.0】"],
  "sections": [
    {
      "id": "【セクションID。例：pci_1】",
      "label": "【セクション表示名。例：要件1. ネットワークセキュリティ管理策の導入と維持】",
      "ref": "【英語の要件名またはカテゴリ名。例：REQUIREMENT 1】",
      "subsections": [
        {
          "title": "【サブセクション名。例：1.1 — ネットワークセキュリティ管理策の定義と理解】",
          "questions": [
            {
              "no": "【質問番号。例：1.1】",
              "text": "【日本語の質問文。文末は「〜している」「〜されている」の体言止めで統一】",
              "refs": ["【識別番号。例：PCI 1.1.1】"]
            }
          ]
        }
      ]
    }
  ]
}
```

---

## 質問文の書き方ルール

1. **文体**：「〜されている」「〜している」「〜を実施している」で統一（体言止め）
2. **粒度**：1問1要件。複数の要件を1問にまとめない
3. **Yes/No で答えられる形式**にする（4択：はい／ほとんど／一部／いいえ で回答）
4. **日本語**で記述する
5. **refs フィールド**：そのフレームワークの識別番号（コントロールID）を配列で指定する
   - 形式：`"【フレームワーク略称】 【番号】"` （例：`"PCI 1.1.1"`、`"ISO 27001 A5.1"`）
   - 1問につき1〜3件程度。関連する識別番号が複数ある場合は複数指定可

---

## 出力形式

- 有効な JSON のみを出力してください（説明文・コードブロック記号は不要）
- 文字コード：UTF-8
- キー順序：スキーマの順序を守ること

---

## 品質チェック（出力前に確認してください）

- [ ] 全セクションに `id`・`label`・`ref`・`subsections` が存在する
- [ ] 全サブセクションに `title`・`questions` が存在する
- [ ] 全質問に `no`・`text`・`refs` が存在する
- [ ] `refs` の値が `"略称 番号"` の形式になっている
- [ ] 質問文が体言止めになっている
- [ ] JSON が構文エラーなく有効である
```

---

## ファイル②：識別番号定義 JSON

```
あなたはサイバーセキュリティの専門家です。
以下の仕様に従い、企業向けセキュリティリスクアセスメントシステム用の識別番号定義 JSON ファイルを作成してください。

---

## 対象フレームワーク

- 名称：【例：PCI DSS v4.0】
- 識別番号一覧：【以下に質問票JSONの refs フィールドに含まれる全識別番号を列挙してください】
  例：
  PCI 1.1.1
  PCI 1.1.2
  PCI 1.2.1
  …（全件）

---

## 出力仕様

以下の JSON スキーマに厳密に従ってください。

```json
{
  "_meta": {
    "version": "1.0",
    "framework": "【フレームワーク正式名称。例：PCI DSS v4.0】",
    "description": "【説明文。例：PCI DSS v4.0 識別番号の概要と詳細定義。〇〇件収録。】",
    "count": 【整数：エントリ件数】
  },
  "【識別番号。例：PCI 1.1.1】": {
    "summary": "【識別番号】 — 【30〜50字程度の日本語概要】",
    "detail": "【正式名称】〇〇\n【目的・概要】\n〇〇\n【主な要件】\n・〇〇\n・〇〇\n・〇〇\n【実装例】\n〇〇"
  }
}
```

---

## 各フィールドの書き方ルール

### summary フィールド
- 形式：`"識別番号 — 日本語の概要"` （ダッシュは全角スペース+半角ハイフン2つ+全角スペース `—` を使用）
- 内容：その管理策が「何をするものか」を30〜50字で端的に表現

### detail フィールド（改行は `\n` で表現）

必ず以下の4セクションを含めること：

```
【正式名称】英語の正式名称（日本語訳を括弧内に入れても可）
【目的・概要】
2〜4文で目的と重要性を説明
【主な要件】
・要件1（箇条書き、3〜5項目）
・要件2
・要件3
【実装例】
具体的なツール名・手順を含む実装例（2〜4文）
```

---

## 出力形式

- 有効な JSON のみを出力してください（説明文・コードブロック記号は不要）
- 文字コード：UTF-8
- 全ての識別番号エントリを省略せず出力すること

---

## 品質チェック（出力前に確認してください）

- [ ] `_meta.count` がエントリ数と一致している
- [ ] 全エントリに `summary`・`detail` が存在する
- [ ] `summary` が `"識別番号 — 概要"` の形式になっている
- [ ] `detail` に【正式名称】【目的・概要】【主な要件】【実装例】の4セクションが全て含まれている
- [ ] 箇条書きが `・` で始まっている
- [ ] JSON が構文エラーなく有効である
```

---

## ファイル③：manifest.json への追記内容

```
以下の内容を manifest.json の "frameworks" 配列の末尾に追加してください。

{
  "id": "【質問票JSONのidと同じ値。例：pci_dss】",
  "label": "【表示名。例：PCI DSS v4.0】",
  "sublabel": "【副題。例：全12要件】",
  "icon": "【絵文字アイコン。例：💳】",
  "color": "【カードの色（16進カラーコード）。例：#dc2626】",
  "description": "【スタート画面のカードに表示される説明文（50〜80字）】",
  "file": "questions/【質問票JSONのファイル名。例：pci_dss.json】",
  "env": null,
  "tags": ["【フレームワーク名。例：PCI DSS v4.0】"],
  "refsFiles": ["content/【定義ファイル名。例：refs_pci.json】"]
}
```

---

## ファイル配置場所（作成後）

| ファイル | 配置先 |
|---|---|
| 質問票 JSON | `questions/【id】.json` |
| 識別番号定義 JSON | `content/refs_【略称】.json` |
| manifest.json | 既存ファイルに追記（`questions/manifest.json`） |

---

## 参考：既存フレームワークの識別番号形式

| フレームワーク | refs の形式 | 例 |
|---|---|---|
| NIST SP 800-53 | `NIST AC-2` | `"NIST AC-2"` |
| CIS Controls v8.1 | `CIS 1.1` | `"CIS 5.3"` |
| CSA CCM v4 | `CSA CCM IAM-06` | `"CSA CCM CEK-01"` |
| ISO/IEC 27001:2022 | `ISO 27001 A5.1` | `"ISO 27001 A8.34"` |
| NIST CSF v2.0 | `CSF GV.OC-01` | `"CSF PR.AA-03"` |

新規フレームワークの識別番号形式は `【略称】 【番号】` の形式で統一してください。
